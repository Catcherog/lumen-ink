# 经验总结机制

> **生效模式**：智能生效
> **触发信号**：任务完成或遇到错误/失败
> **优先级**：在 `_memory.md` 写入之后执行（先写记忆，再总结经验）
> **前置依赖**：若触发此规则，`_memory.md` 的写入规则同步生效

---

## 触发判定

| 判定条件 | 是否触发 | 示例 |
|---------|---------|------|
| 完成涉及调试的任务 | **触发** | 修复了一个 API 调用错误 |
| 完成涉及配置/集成的任务 | **触发** | 配置了飞书 API 认证 |
| 完成涉及架构决策的任务 | **触发** | 选择了 A 方案而非 B 方案 |
| 遇到报错并解决 | **触发** | npm install 失败后找到解决方案 |
| 方案不可行需要换方向 | **触发** | 某个 MCP 工具不支持所需操作 |
| 简单的代码修改 | **不触发** | 修改一个变量名 |
| 纯信息查询/回答 | **不触发** | "这个函数是做什么的？" |
| 文件创建/移动 | **不触发** | 仅触发 `_file_management.md` |

## 复杂任务判定阈值

满足以下**任一**条件即为复杂任务：
1. 任务涉及 3 个以上工具调用（Read + Edit + RunCommand 等）
2. 任务执行过程中遇到过错误并需要修复
3. 任务涉及跨模块/跨系统的配置或集成
4. 用户明确要求"总结一下"或"记录一下"

## 执行逻辑

```
任务完成 / 踩坑
  ↓
判定是否复杂任务 → 是：继续 / 否：不触发
  ↓
提取关键信息 → 做了什么？关键决策？遇到的坑？解决方案？
  ↓
选择类型 → EXPERIENCE / LESSON / DECISION
  ↓
按格式写入 project_memory.md
  ↓
[可选] 提醒用户是否需要更详细的归档
```

## 记录格式

```
[EXPERIENCE] 标题 | YYYY-MM-DD
- 场景：什么情况下做的
- 方案：采取了什么行动
- 结果：效果如何

[LESSON] 标题 | YYYY-MM-DD
- 问题：遇到了什么错误/失败
- 原因：根本原因是什么
- 解决：最终怎么解决的

[DECISION] 标题 | YYYY-MM-DD
- 背景：为什么需要做决策
- 选项：有哪些候选方案
- 决策：选择了什么，为什么
```

## 边界情况

| 情况 | 处理方式 |
|------|---------|
| 任务部分完成（未结束） | 不触发，等任务完全结束再总结 |
| 同一会话多次踩坑 | 每次踩坑单独记录一条 LESSON |
| 用户中断任务 | 不触发总结，但若已踩坑则记录 LESSON |
| 经验与已有记录重复 | 合并到已有条目，不重复创建 |
| project_memory.md 不存在 | 先创建文件再写入 |

## 定期回顾

- **触发条件**：会话开始时，若 `project_memory.md` 中存在超过 7 天未回顾的条目
- **执行方式**：向用户提示未回顾条目的标题列表
- **用户操作**：确认保留 / 归档 / 更新 / 删除
